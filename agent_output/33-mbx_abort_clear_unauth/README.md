Vulnerability: mbx hostif_control_abort_clear_i used without authorization (CWE-284)
Module: mbx
RTL file: hw/ip/mbx/rtl/mbx_imbx.sv
Vulnerability ID: 33

Summary
-------
The mbx inbound mailbox RTL uses the raw hostif_control_abort_clear_i signal as an unauthenticated control in multiple critical expressions without state or ownership checks. Specifically, it is used to form host_clear_abort (hostif_control_abort_clear_i & mbx_sys_abort), is OR'd into load_write_ptr, and directly clears pending via clear_pending = hostif_sram_write_gnt_i | hostif_control_abort_clear_i. An unprivileged MMIO writer that can toggle the abort_clear bit can therefore clear pending, acknowledge aborts, and cause write pointer reloads without proper authorization.

Exploit Type
------------
POC demonstrating improper access control: an unprivileged MMIO write to the abort_clear control can clear pending state in the MBX inbound logic.

APIs used
---------
- abs_mmio_write32() - low-level MMIO write to model unprivileged writes
- mmio_region_from_addr(), mmio_region_read32(), mmio_region_write32() - structured MMIO access
- bitfield helpers for bit checks
- OTTF test framework for assertions and logging

What the POC does
-----------------
1. Simulates a pending inbound message by setting the STATUS.pending bit.
2. Performs an unprivileged write to the abort_clear control bit (hostif_control_abort_clear_i) using abs_mmio_write32().
3. Observes that STATUS.pending is cleared as a result of that write, proving that the control bit can unilaterally clear pending without authorization.
4. Reads the write pointer before and after an abort_clear write to demonstrate potential pointer reload effects.

How to build and run
--------------------
1. Build with Bazel:
   bazel build //path/to:mbx_vuln_test
2. Run tests in the simulation environment (Verilator or DV), or on hardware if appropriate:
   bazel test //path/to:mbx_vuln_test --test_output=all

Notes and Limitations
---------------------
- The POC uses symbolic register offsets (MBX_CTRL_REG_OFFSET, MBX_STATUS_REG_OFFSET, etc.) that should correspond to the real autogenerated register layout (available via the mbx_c_regs target). The BUILD target depends on //hw/top:mbx_c_regs and //hw/top_earlgrey/sw/autogen:top_earlgrey to provide the correct headers and base addresses.
- The exact behavior of write-pointer reload depends on the external mbx_sys_abort signal. The POC demonstrates the unauthorized ability to assert abort_clear and observe side effects; an attacker controlling an MMIO write can therefore cause DoS, data corruption, and flow-control bypass as described in the vulnerability report.
- No DIF APIs are used; only base library primitives are used to model low-level MMIO writes and observations.
